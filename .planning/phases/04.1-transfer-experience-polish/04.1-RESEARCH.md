## Summary

Phase 4 delivered all six CC-01 through CC-06 requirements and passed verification (6/6). However, a real browser test session revealed concrete UX gaps, at least one confirmed bug that causes a blank-screen crash, and several polish items that were explicitly deferred as "Claude's discretion" during Phase 4. Phase 04.1 addresses these gaps without introducing new infrastructure — all work is frontend-only React component polish and minor server-side validation fixes.

The four distinct areas of work are: (1) fixing the confirmed blank-screen crash when a transfer user is created with no transcript courses; (2) improving the dashboard layout so the three progress cards are visually clear and the CC/UCSB course pairing is explicit in the major lower-div card; (3) adding a "change target major" affordance so transfer students can update their target major after onboarding; and (4) implementing a UCSB transition flow — a one-click action that converts a transfer student account into a UCSB student account once they have been admitted.

All Phase 4 infrastructure (hooks, utilities, data files, Supabase schema) remains unchanged. This phase is entirely UI-layer work.

**Primary recommendation:** Fix the crash first (it is a hard blocker for any demo), then tackle dashboard layout polish, then the target major change flow, then the UCSB transition flow. No new dependencies. No Supabase schema changes.

## Architecture Patterns

### student_type Branching (established)
All transfer-specific rendering already uses early-return branching at the DashboardView/RoadmapView/WhatIfView level. Phase 04.1 follows the same pattern — no new top-level routes.

### Mutable User Fields via PUT /api/users/:id
The server's PUT handler does a full merge: `users[index] = { ...users[index], ...req.body, id: users[index].id }`. This means any field on the user object can be updated by calling:

```js
await fetch(`/api/users/${user.id}`, {
  method: 'PUT',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ target_major_id: newMajorId }),
});
```

This supports both the target major change flow (POLISH-04) and the UCSB transition flow (POLISH-05) without any server changes.

### UCSB Transition Flow — State Model

A transfer student transitioning to UCSB needs to:
1. Change `student_type` from `'transfer'` to `'ucsb'`
2. Set `school: 'University of California Santa Barbara (UCSB)'`
3. Set `major: majorName` (the human-readable name from MAJOR_CONFIGS)
4. Set `majorId: target_major_id` (the key from MAJOR_CONFIGS, e.g. `'cs_bs'`)
5. Retain their existing transcript (completed CC courses remain as history)
6. Navigate the UI to the UCSB dashboard

No Supabase changes needed. No new server endpoints. One PUT request with the above fields.

After the transition, `DashboardView.jsx` will route to the UCSB branch (it already checks `user?.student_type === 'transfer'`). The UCSB dashboard uses `requirements` which is derived from `MAJOR_CONFIGS[user.majorId]`. The user's CC transcript courses will not match UCSB requirement IDs, so the dashboard will show 0% progress initially — which is correct (they are starting UCSB).

**Where to trigger the transition:**
- A "I've been admitted to UCSB" button/section on the TransferDashboardView or TransferRoadmapView is the natural place.
- Should show a confirmation modal (irreversible action changes the experience).

### Target Major Change Flow — State Model

The user has `target_major_id` on their profile. To change it:
1. Show a `<select>` with all 12 major options (the same `majorOptions` array already in ProfileWizard)
2. On confirm: `PUT /api/users/:id` with `{ target_major_id: newId }`
3. Update `currentUser` state in CompassDemo.jsx

**Where to place the UI:**
- Dashboard header area (next to "Track your transfer progress toward...") as an inline edit
- OR a settings icon/button in the sidebar for the current user menu

The simplest placement: an "Edit" button next to the target major name in the TransferDashboardView welcome header, showing a dropdown on click.

## Common Pitfalls

### Pitfall 1: UCSB Transition Leaves Broken State
**What goes wrong:** After transition, the UCSB dashboard tries to read `requirements` from `MAJOR_CONFIGS[user.majorId]`. If `majorId` is not set on the user (only `target_major_id` exists for transfer users), requirements will be undefined, crashing the dashboard.
**How to avoid:** The transition PUT request must set BOTH `majorId: target_major_id` AND `major: MAJOR_CONFIGS[target_major_id].name`. Verify that CompassDemo.jsx reads `user.majorId` (not `user.target_major_id`) for UCSB path. Check `getMajorId()` in CompassDemo.jsx — it likely prefers `user.majorId`.
**Warning signs:** Dashboard shows 0 requirements or crashes after transition.

### Pitfall 2: theme-usage.test.js on Any New Components
**What goes wrong:** Any new modal or settings panel that uses raw hex strings fails the automated test.
**How to avoid:** Same constraint as Phase 4 — use `theme.colors.*` exclusively in all inline styles.

### Pitfall 3: Transfer User Sidebar Subtitle Breaks for Existing Users
**What goes wrong:** The sidebar subtitle fix for transfer users needs to handle the case where `institutions` data is not yet loaded (async Supabase fetch) or where `source_institution_id` is a UUID that is not in the local institutions list.
**How to avoid:** Use a fallback chain: institution name from loaded institutions data → target major name from MAJOR_CONFIGS → literal "Transfer Student". Do not block rendering on the Supabase call.

### Pitfall 4: Transition is Irreversible but Has No Confirmation
**What goes wrong:** User accidentally clicks "I've been admitted" and loses the transfer experience — no way to undo.
**How to avoid:** Show a modal with a clear warning: "This will switch your account to UCSB student mode. Your completed courses will be kept as history." Require an explicit confirmation button, not just a single click.

### Pitfall 5: Target Major Change Doesn't Propagate to TransferWhatIfView
**What goes wrong:** After changing `target_major_id`, the TransferWhatIfView highlights the OLD major as current target because it reads `user.target_major_id` from the prop. If `currentUser` state in CompassDemo.jsx is updated immediately, this works. If the UI only updates on next server fetch, there's a stale display.
**How to avoid:** After a successful `PUT /api/users/:id`, update the local `currentUser` state immediately in CompassDemo.jsx (same pattern as other user field updates). Do not wait for a re-fetch.

### Pitfall 6: Dashboard 3-Column Grid on Narrow Screens
**What goes wrong:** The `repeat(3, 1fr)` grid at `TransferDashboardView.jsx` line 130 works fine on desktop but the IGETC card has a `maxHeight: 220px` overflow scroll. If the card content is very long (10 areas), the card height will be inconsistent with the other two cards.
**How to avoid:** Set a `minHeight` on all three cards, or use `align-items: stretch` (already the default for grid). The overflow scroll is the right choice for the IGETC list — just ensure the card header+ring+list layout is visually balanced. This is a "Claude's discretion" styling matter.

## Code Examples

### Target Major Change — Inline Edit Pattern

```jsx
// In TransferDashboardView welcome header
const [editingMajor, setEditingMajor] = useState(false);
const [pendingMajorId, setPendingMajorId] = useState(user?.target_major_id ?? '');

const handleMajorSave = async () => {
  if (!pendingMajorId || pendingMajorId === user.target_major_id) {
    setEditingMajor(false);
    return;
  }
  const res = await fetch(`/api/users/${user.id}`, {
    method: 'PUT',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ target_major_id: pendingMajorId }),
  });
  if (res.ok) {
    const updated = await res.json();
    onUserUpdate(updated); // callback to update currentUser in CompassDemo.jsx
    setEditingMajor(false);
  }
};
```

## Open Questions

1. **Where to place the UCSB transition trigger**
   - What we know: Must be visible but not accidentally clicked; needs a confirmation step
   - Options: (a) A section at the bottom of TransferDashboardView titled "Ready to Transfer?" with a CTA button; (b) A settings option in the Sidebar user menu
   - Recommendation: Add it as a section on TransferDashboardView (bottom of page, after Educational History) — more discoverable for demo purposes. Claude's discretion applies.

2. **Where to place the Change Target Major affordance**
   - What we know: The target major appears in the dashboard welcome header subtitle and in the Roadmap header
   - Recommendation: Inline edit on the TransferDashboardView welcome header — clicking the major name opens a `<select>` with a Save button. This is the least disruptive placement. Claude's discretion applies.

3. **Should the Sidebar import MAJOR_CONFIGS for the subtitle fix?**
   - What we know: Sidebar currently has no data dependencies; adding MAJOR_CONFIGS is a small but real coupling change
   - Recommendation: Pass a computed `subtitle` prop from CompassDemo.jsx rather than adding a MAJOR_CONFIGS import to Sidebar. This maintains Sidebar's purity as a presentational component.

4. **knownCourses prop for transfer student's current-quarter autocomplete**
   - What we know: `DashboardView.jsx` passes `knownCourses` (built from the UCSB major's requirements) to `TransferDashboardView`. For transfer students, these are UCSB course IDs, not CC course IDs — so the autocomplete suggestions are irrelevant.
   - What's unclear: Whether this causes a visible bug or just shows no suggestions (because CC course IDs like "MATH 3A" won't match UCSB requirement IDs)
   - Recommendation: For transfer students, the current-quarter autocomplete should suggest from CC articulation `source_course_code` values. TransferDashboardView already has `articulations` loaded — it can derive CC course suggestions from `articulations.map(a => a.source_course_code)` instead of using the `knownCourses` prop. This is a discrete improvement, not a blocker.

5. **ProfileWizard res.ok guard — is it present?**
   - What we know: Debug file says it was missing; server.js fix means a 400 won't happen in normal flow; the guard may still be missing for edge cases
   - Recommendation: Add an explicit `if (!res.ok) { console.error(...); return; }` guard to `handleTransferComplete` regardless of whether the server fix is applied. Defense in depth.
